# 2026年度 写真展サイト・アンケート運用マニュアル

最終更新: 2026年7月

## 1. このマニュアルの目的

このマニュアルは、法政大学 小金井写真部の幹部が、写真展サイトと来場者アンケートを安全に引き継いで運用するためのものです。

対象となる作業は次のとおりです。

- 出展者から作品掲載の同意を得る
- 公開用画像と作品情報を集める
- GitHub Pagesへ作品一覧・アンケートページを公開する
- Supabaseで回答受付を開始・停止する
- 回答を確認・集計する
- 会期終了後に作品と回答データを整理する

本書は部内の実務ルールです。個別の作品で著作権、肖像権、施設の撮影条件等に疑問がある場合は、無理に掲載せず、大学の担当窓口や専門家へ確認してください。

## 2. 最初に理解しておくこと

### 2.1 GitHub PagesとSupabaseの役割

| サービス | 用途 | 保存するもの |
| --- | --- | --- |
| GitHub Pages | 作品表示とアンケート画面 | 公開用作品画像、作品情報、画面プログラム |
| Supabase | 回答受付と保存 | 選択作品、作品への感想、写真展全体への感想、送信日時 |
| 部内ストレージ | 原本の保管 | 高解像度画像、掲載同意記録、集計後の管理資料 |

GitHub Pagesは公開サイトです。パスワード画面、長いURL、QRコード限定配布は、閲覧者を限定する本格的な認証ではありません。

### 2.2 公開リポジトリへ原本を置かない

公開リポジトリへ一度コミットした画像は、現在のページから削除してもGit履歴に残る可能性があります。そのため、次を必ず守ります。

- カメラから取り出した原本を置かない
- 印刷用・提出用の高解像度画像を置かない
- Web掲載用に縮小・圧縮した派生画像だけを置く
- EXIF位置情報等のメタデータを削除する
- 掲載同意が確認できない画像を置かない

部員限定アーカイブが必要な場合は、GitHub Pagesではなく、大学または部で管理する認証付きストレージを利用します。

### 2.3 著作権は作者に残る

サイト掲載の同意は、著作権を写真部へ譲渡してもらう手続きではありません。作者から、決められた目的・期間・方法で掲載する許可を得るものです。

文化庁も、著作者には作品の利用を許諾または禁止する権利があること、他人の著作物を利用する際には著作者人格権、肖像権、利用条件等も確認する必要があることを案内しています。

- [著作権の基本と海賊版（文化庁）](https://www.bunka.go.jp/seisaku/chosakuken/kaizoku/kihon.html)
- [他人の著作物を利用したい場合など（文化庁）](https://www.bunka.go.jp/seisaku/chosakuken/seidokaisetsu/chosakukensha_fumei/)
- [著作権契約書作成支援システム（文化庁）](https://pf.bunka.go.jp/chosaku/chosakuken/c-template/)

## 3. 運用担当者と権限

1人へ権限を集中させず、最低2人で確認します。

| 役割 | 主な担当 | 必要な権限 |
| --- | --- | --- |
| 運用責任者 | 日程、掲載同意、公開判断、緊急停止 | GitHub閲覧、Supabase管理 |
| Web担当 | 画像加工、作品データ、ページ作成、デプロイ | GitHub書込 |
| 回答管理担当 | 受付開始・停止、CSV出力、集計、削除 | Supabase管理 |
| 確認担当 | 表示、作者名、作品番号、リンク、同意範囲の照合 | GitHub閲覧 |

推奨ルール:

- GitHubとSupabaseは個人ごとのアカウントを使う
- パスワードを共有しない
- 多要素認証を有効にする
- 退任者の権限は引継ぎ完了後すぐ削除する
- Secret key、`service_role` key、Database passwordをGitHubへ保存しない

## 4. 年間の標準スケジュール

### 会期6〜8週間前

- 展示名、会期、回答受付期間を決定
- Web掲載の有無を出展要項へ明記
- 出展者へ掲載同意確認を配布
- 掲載を希望しない作者の作品を管理表へ記録

### 会期3〜4週間前

- 掲載同意済み作品だけを回収
- 作品番号、タイトル、作者表示名を確定
- 肖像、他人の著作物、施設の撮影・公開条件を確認
- Web用画像を作成

### 会期1〜2週間前

- 作品YAMLとアンケートページを作成
- Supabaseへ展示情報を登録
- 回答受付を停止した状態でGitHub Pagesへ公開
- PCとスマートフォンで確認
- QRコードを作成

### 会期前日

- 作者名、作品番号、画像の対応を2人で照合
- 掲載不同意作品が含まれていないことを確認
- QRコードのリンク先を実機で確認
- Supabaseの回答件数が0件であることを確認
- 緊急連絡先と停止担当者を共有

### 会期開始

- Supabaseで受付を有効化
- テスト回答を1件送信
- 保存確認後、テスト回答を削除
- 会場QRを掲示

### 会期終了直後

- 最初に回答受付を停止
- 回答件数を記録
- CSVを出力して部内の管理場所へ保存
- 会場QRを撤去

### 会期終了後

- 作者ごとの同意期間に従って一般公開を終了
- 公開対象から画像を削除
- 集計結果を作成
- 不要になった自由記述・回答データを削除
- 次年度担当者向けに問題点を記録

## 5. 作品掲載同意

### 5.1 同意は作品ごとに記録する

出展申込みとWeb掲載同意を同一視しません。最低限、次を作者へ示し、記録を残します。

- 対象作品
- 掲載サイトと目的
- 一般公開されること
- 掲載予定期間
- 画像の縮小・圧縮・透かし等
- 表示する作者名
- SNSや広報物への二次利用の有無
- 会期後アーカイブの有無
- 連絡先と掲載取下げ方法

[作品掲載同意確認のひな形](./2026作品掲載同意確認ひな形.md)を利用できます。

### 5.2 チェックが必要な作品

次に該当する場合は、作者本人の著作権だけで掲載可否を判断しません。

- 人物の顔や個人を特定できる情報が写っている
- 絵画、ポスター、キャラクター、舞台、展示物等が主要被写体になっている
- 私有地、店舗、美術館、ライブ会場等で撮影している
- 撮影規約でWeb公開やコンテスト応募が制限されている
- 共同制作、借用品、生成AI素材、第三者素材を使用している

人物が写る場合は、作者に被写体の同意確認を求めます。確認できない場合は、トリミングやぼかしで十分かを検討し、判断できなければ掲載しません。文化庁も、他人の肖像が含まれる場合には肖像権等への注意を案内しています。

- [他人の肖像や著作物は含まれていませんか（文化庁）](https://www.bunka.go.jp/jiyuriyo/chui3.html)

### 5.3 掲載しない判断を尊重する

- Web掲載を断っても展示参加や審査で不利益を与えない
- 同意が曖昧な場合は「掲載しない」と判断する
- SNS掲載、サイト掲載、会期後アーカイブは別々に確認する
- 作者から取下げ依頼が来た場合は、運用責任者とWeb担当が速やかに対応する

## 6. 公開用画像の作成基準

年度の途中で基準を変えず、展示ごとに統一します。推奨初期値は次のとおりです。

- 長辺: 1600px程度
- 形式: JPEGまたはWebP
- 画質: 画面鑑賞に十分だが印刷原稿には不向きな品質
- メタデータ: EXIF、GPS、カメラ所有者名を削除
- ファイル名: `2026-summer01_s.jpg`のように展示と作品番号を含める
- 色: sRGB
- 透かし: 作者との合意がある場合のみ統一形式で付与

画像保存防止の右クリック無効化や透かしは、無断利用を完全には防げません。「保存できない」と作者へ説明しないでください。

### Web公開用画像へのロゴ透かし

Web公開用画像には、写真部ロゴを白色・不透明度8%・斜め反復配置で焼き込みます。原本を直接加工せず、Web用画像から`_wm`付きの別ファイルを生成します。

```sh
scripts/watermark_web_images.sh \
  2025/gakusai/<公開用ディレクトリ>/public-images \
  images/photoiconclubs.png
```

この処理はJPEG品質88で書き出し、EXIF等のメタデータを除去します。生成後は、画像の向き、作品番号、ロゴの見え方、YAMLの参照先を2人で確認してください。透かしは無断利用を困難にするための抑止策であり、スクリーンショットや保存を完全に防止するものではありません。

## 7. 2026年度展示の追加手順

以下は`2026-summer`を例にしています。学祭なら`2026-gakusai`、冬展なら`2026-winter`へ置き換えます。

### 7.1 作品データを作る

`_data/works/2026-summer.yml`を作成します。

```yaml
- "no": 1
  title: "作品タイトル"
  author: "公開する作者名"
  camera: ""
  lens: ""
  location: ""
  shot_date: ""
  images:
    thumb: "/2026/summer/public-images/2026-summer01_s.jpg"
    full: "/2026/summer/public-images/2026-summer01_s.jpg"
  description: |
    作者による作品説明
```

作品番号は重複させず、画像ファイルとの対応を2人で確認します。

### 7.2 アンケートページを作る

`2026/summer/survey.html`を作成します。

```html
---
layout: survey
title: "2026年度 夏写真展アンケート"
exhibition_key: "2026-summer"
works_data: "2026-summer"
readonly_url: "/2026/summer/works/"
entry_url: "/2026/summer/"
---

<header class="survey-intro">
  <h1>2026年度 夏写真展アンケート</h1>
  <p>お気に入りの作品を1〜3作品選び、ご感想をお寄せください。</p>
  <p>氏名、連絡先などの個人情報は入力しないでください。</p>
</header>
```

### 7.3 QR入口ページを作る

QRコードはアンケートの直URLではなく、展示ごとの固定入口を指すようにします。`2026/summer/index.html`の例です。

```html
---
layout: exhibition-entry
title: "2026年度 夏写真展"
exhibition_key: "2026-summer"
survey_url: "/2026/summer/survey/"
works_url: "/2026/summer/works/"
---

<h1>2026年度 夏写真展</h1>
<p>法政大学 小金井写真部</p>
```

Supabaseのサーバー時刻により、開始前、会期中、終了後の画面が自動で切り替わります。終了後に部員限定アーカイブへ自動転送してはいけません。

`exhibition_key`は、作品データ、HTML、Supabaseで完全に一致させます。

### 7.4 Supabaseへ展示を登録する

作品が1〜40番、回答期間が2026年9月12日10時から9月14日18時の場合の例です。

```sql
insert into public.survey_exhibitions (
  exhibition_key,
  title,
  work_ids,
  opens_at,
  closes_at,
  is_accepting_responses
) values (
  '2026-summer',
  '2026年度 夏写真展',
  array(select n::text from generate_series(1, 40) n),
  '2026-09-12 10:00:00+09',
  '2026-09-14 18:00:00+09',
  false
)
on conflict (exhibition_key) do update
set title = excluded.title,
    work_ids = excluded.work_ids,
    opens_at = excluded.opens_at,
    closes_at = excluded.closes_at,
    is_accepting_responses = false;
```

連番でない場合は、次のように明示します。

```sql
array['1', '2', '4', '5']
```

### 7.5 公開前チェック

- [ ] 全作品で作者の掲載同意が確認できる
- [ ] 画像はWeb用派生画像である
- [ ] EXIF・GPSを削除した
- [ ] 作品番号、画像、タイトル、作者名が一致している
- [ ] 掲載不同意作品が含まれていない
- [ ] PCとスマートフォンで表示できる
- [ ] 1作品、2作品、3作品の選択を確認した
- [ ] 3作品選択時に他のボタンが無効になる
- [ ] 確認、編集、送信完了を確認した
- [ ] 再アクセス時に閲覧専用ページへ移る
- [ ] `is_accepting_responses`は公開確認中も`false`である
- [ ] QRが固定入口ページを指している
- [ ] 開始前画面、会期中画面、終了画面をテストした
- [ ] アンケート直リンクも開始前・終了後に入力画面を表示しない

## 8. 回答受付の開始と停止

### 開始

日時設定が正しいことを確認してから実行します。

```sql
update public.survey_exhibitions
set is_accepting_responses = true
where exhibition_key = '2026-summer';
```

### 停止

会期終了時または問題発生時に実行します。

```sql
update public.survey_exhibitions
set is_accepting_responses = false
where exhibition_key = '2026-summer';
```

### 状態確認

```sql
select exhibition_key, opens_at, closes_at, is_accepting_responses
from public.survey_exhibitions
where exhibition_key = '2026-summer';
```

公開期間の判定はSupabase側でも行われるため、`is_accepting_responses = true`でも`opens_at`より前、または`closes_at`より後は送信できません。

## 9. 回答データの扱い

### 9.1 取得する情報を増やさない

現在のアンケートでは、氏名、メールアドレス、学籍番号、年齢、性別等を尋ねません。必要性が明確でない項目を追加しないでください。

自由記述には個人情報が入力される可能性があるため、アンケート画面には「氏名・連絡先等を入力しないでください」と表示します。

個人情報を取得する場合は、取得前に利用目的を分かりやすく表示します。個人情報保護委員会のガイドラインでも、アンケート等で個人情報を取得する際の利用目的の明示・公表が示されています。

- [個人情報保護法ガイドライン（通則編）](https://www.ppc.go.jp/personalinfo/legal/guidelines_tsusoku/)

### 9.2 利用目的

回答は次の目的に限定します。

- 来場者が選んだ作品の集計
- 写真展の改善
- 部内での振り返り
- 個人を特定しない統計結果の作成

自由記述をそのままSNS、Webサイト、部誌等へ転載しません。掲載したい場合は、アンケート上で引用可否を別途確認する仕組みを用意します。

### 9.3 閲覧できる人

- Supabaseの回答閲覧権限は運用責任者と回答管理担当に限定
- CSVを部の公開チャットへ投稿しない
- 集計作業用ファイルを個人の端末へ放置しない
- 引継ぎ後、不要になった前任者の権限を削除

### 9.4 保存期間の部内方針

展示ごとに保存期限を決め、同意説明またはアンケート案内へ記載します。推奨方針は次のとおりです。

- 生の回答と自由記述: 集計確定後90日以内に削除
- 個人を特定しない集計結果: 部内記録として保存可
- テスト回答: 本番開始前に削除

保存する目的がなくなったデータを「念のため」で残し続けないことが重要です。

## 10. 集計

回答数と選択作品を確認する基本SQLです。

```sql
select
  s.work_id,
  count(*) as votes
from public.survey_response_selections s
join public.survey_responses r on r.id = s.response_id
join public.survey_exhibitions e on e.id = r.exhibition_id
where e.exhibition_key = '2026-summer'
group by s.work_id
order by votes desc, s.work_id;
```

全体の回答者数を確認します。

```sql
select count(*) as responses
from public.survey_responses r
join public.survey_exhibitions e on e.id = r.exhibition_id
where e.exhibition_key = '2026-summer';
```

投票数と回答者数は異なります。1人が最大3作品を選べるため、結果発表では「回答者数」と「作品ごとの得票数」を混同しないでください。

## 11. テスト回答の削除

本番開始前に、展示キーを十分確認してから実行します。この操作は元に戻せません。

```sql
delete from public.survey_responses r
using public.survey_exhibitions e
where r.exhibition_id = e.id
  and e.exhibition_key = '2026-summer';
```

選択作品の行は外部キーの`on delete cascade`により同時に削除されます。必ず回答受付を停止し、対象展示を確認してから実行します。

## 12. トラブル対応

### 回答を停止したい

最優先で`is_accepting_responses = false`にします。GitHub Pagesの修正より先にSupabaseを停止します。

### 間違った画像・作者名を公開した

1. アンケート受付を停止
2. 問題のページへの案内を止める
3. 作者へ連絡
4. GitHubで修正または画像を削除
5. Pagesの再デプロイを確認
6. 原因と対応を部内記録へ残す

### 作者から掲載取下げ依頼が来た

同意記録と本人確認を行い、対象画像・表示を速やかに停止します。Git履歴からの完全削除が必要な事情がある場合は、通常のファイル削除だけで済ませず、リポジトリ管理者と対応を協議します。

### 回答できない

- `is_accepting_responses`
- `opens_at`と`closes_at`
- `work_ids`
- GitHub Pagesの最新デプロイ
- Supabaseの稼働状況

を確認します。

### 誤ってSecret keyを公開した

1. 直ちにSupabaseで該当キーを失効・再発行
2. 回答受付を停止
3. Git履歴を含めて漏えい範囲を確認
4. 不審な操作や回答データへのアクセスを確認
5. 部内責任者へ報告

ファイルを削除するだけでは対応完了になりません。

## 14. 重要な運用原則

迷った場合は、次の順番で判断します。

1. 作者と来場者の権利を守る
2. 回答受付を止めて被害拡大を防ぐ
3. 同意記録と設定を確認する
4. 2人以上で公開・再開を判断する
5. 起きたことと対応を記録し、次年度へ引き継ぐ
